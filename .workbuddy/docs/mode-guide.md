## 一、三种模式速查

### Ask 模式（问一问）—— 军师，只说不做

| 属性 | 说明 |
|------|------|
| **别名** | 问一问、顾问模式 |
| **权限** | 只读，不修改任何文件 |
| **速度** | 最快 |
| **积分消耗** | 最少 |
| **定位** | 消除不确定性的第一步 |

**典型场景**：
- 查语法、查 API 用法、解释代码
- 搜索知识点、确认概念
- 对 AI 的能力有疑问时先问清楚
- 讨论方案、理清思路

**使用口诀**：不确定就先问。

### Plan 模式（想一想）—— 项目经理，先列计划

| 属性 | 说明 |
|------|------|
| **别名** | 想一想、规划模式 |
| **权限** | 可读文件，制定计划但不执行 |
| **速度** | 中等 |
| **积分消耗** | 中等 |
| **定位** | 把模糊需求变成清晰步骤 |

**典型场景**：
- 接到模糊需求，需要先梳理整体思路
- 技术方案选型、架构设计、方案对比
- 多步骤复杂任务（竞品分析、市场调研报告、产品发布会策划）
- 不确定 AI 执行路径的任务

**使用口诀**：超过两步就用 Plan。

### Craft 模式（做一做）—— 执行者，直接开干

| 属性 | 说明 |
|------|------|
| **别名** | 做一做、执行模式 |
| **权限** | 最高，可读写文件、执行代码 |
| **速度** | 较慢（步步确认） |
| **积分消耗** | 最高 |
| **定位** | 交付可用的成品 |

**典型场景**：
- 写代码、脚本、配置文件
- 生成结构化文档、技术方案、报告
- 批量处理文件（改名、合并、格式转换）
- 目标明确的具体操作

**使用口诀**：目标明确就开干，但复杂任务先用 Plan。

## 二、模式选择决策树

```
收到一个任务
    │
    ├─ 只是查资料/讨论？ ────→ ASK
    │
    ├─ 不确定怎么做？ ────→ ASK（先问清楚）
    │                          ↓
    │                     需要动手？
    │                          ↓
    ├─ 超过2步？/不确定步骤？ ──→ PLAN（先出方案）
    │                               ↓
    │                          方案确认？
    │                               ↓
    └─ 目标明确/步骤清楚？ ────→ CRAFT（直接执行）
```

## 三、Ask → Plan → Craft 串联工作流

### 核心公式

```
ASK（澄清信息）→ PLAN（设计方案）→ CRAFT（执行交付）
  快+省             中               慢+贵
```

### 实操流程

**第 1 步：Ask — 消除不确定性**
> "这个需求我理解是 X，但 A/B/C 几点需要确认..."
> 把需求问清楚，搞清楚边界条件和约束。

**第 2 步：Plan — 拿出执行方案**
> "基于以上讨论，我建议分 3 步走：1)... 2)... 3)... 你看看？"
> 把方案拆成清晰步骤，让用户确认方向。

**第 3 步：Craft — 直接交付成果**
> "方案确认，开始执行..."
> 按方案执行，步步推进，直到交付成品。

### 积分优化案例

| 场景 | 错误做法 | 正确做法 | 节省 |
|------|---------|---------|------|
| 整理 50 个 Excel | 直接 Craft 18min/380积分 | Plan→Craft 5min/70积分 | **81%积分，72%时间** |
| 写竞品分析 | 直接 Craft 逐项生成 | Ask(需求)→Plan(框架)→Craft(填充) | 显著减少返工 |
| 搭建新功能 | 边想边写反复改 | Plan(架构设计)→Craft(按方案写) | 避免推倒重来 |

**核心原则：越复杂的任务，越需要 Plan 先行。**

## 五、新任务模式选择规则

### 规则 1：优先 Ask
任何不确定的需求，先用 Ask 模式问清楚。不要一上来就 Craft。

### 规则 2：两步法则
任何涉及 2 步以上的操作，必须经过 Plan 模式。

### 规则 3：Craft 收割
只有目标明确、步骤清楚、不需要讨论的任务，才直接用 Craft。

### 规则 4：Plan 降本
如果预估任务复杂、可能返工，Plan 模式能显著降低积分消耗。

## 六、常见错误

| 错误 | 后果 | 纠正 |
|------|------|------|
| 所有任务都用 Craft | 积分浪费、容易返工 | 按决策树选择 |
| 从不切 Plan | 复杂任务缺少规划、执行混乱 | 超过 2 步必 Plan |
| 在 Ask 里要求执行操作 | Ask 无法操作文件 | 问清楚后切 Craft |
| Plan 之后不确认就执行 | 偏离用户预期 | Plan 结果必须确认 |

## 七、指令模板

### 快速 Ask
```
"帮我查一下 XXX 是什么"
"解释一下这段代码的逻辑"
"XXX 和 YYY 有什么区别"
```

### 标准 Plan
```
"帮我规划一个 XXX 方案，包括步骤清单和预期产出"
"分析一下 XXX，给出分步执行建议"
"对比 A 和 B 两种方案，推荐最优路线"
```

### 高效 Craft（基于 Plan 确认后）
```
"按照刚才确认的方案，开始执行"
"基于 Plan 输出的 3 个步骤，依次完成"
```
